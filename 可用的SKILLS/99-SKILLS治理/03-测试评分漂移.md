# 测试评分与漂移

## 三类测试

### 1. 触发测试

验证是否选对 Skill：

- 正例：应该触发。
- 负例：不应该触发。
- 近似负例：有关键词，但语义不属于该 Skill。

### 2. 行为测试

验证触发后是否按正确流程做：

- 是否先分类？
- 是否区分业务错误和 WAF？
- 是否区分 token 生成和业务接口接受？
- 是否保留 314 框架边界？
- 是否有测试闭环？

### 3. 漂移测试

验证时间变化后是否退化：

- 网站接口变化。
- WAF challenge 变化。
- 用户说法变化。
- Skill description 改动。
- 新增 eval 后旧场景是否变差。

## 评分建议

| 维度 | 权重 |
|---|---:|
| 触发准确率 | 25% |
| 真实任务完成率 | 30% |
| eval 覆盖率 | 20% |
| 可维护性 | 15% |
| 漂移稳定性 | 10% |

## 每次真实任务后必须新增什么

至少新增一个：

- reference 案例
- eval 正例
- eval 负例
- 失败分类
- 触发词
- 版本变更记录

## 官方 Skill Bench

本地结构校验只说明 Skill 格式可用。官方 Skill Bench 跑分需要：

- 仓库可见路径，例如 `skills/<skill-name>/`
- GitHub Actions workflow
- API key secret，例如 `ANTHROPIC_API_KEY`
- PR 或定时触发

没有官方 CI 时，也要保留本地人工评分和 eval 记录。
