# 一致性验证规约 (Consistency Verification Spec)

逆向产出**必须**经过一致性验证才能称为"完成"。本文件定义验证机制、文件格式、阈值、CI 规则。

## 为什么需要

结构完美 + 流程清晰 ≠ 接口正确。任何 Skill 都能跑出"语法对、字段全、API 200"但**数据和真实网页对不上**的结果。逆向交付必须证明:**用我的接口跑出来的数据,和用浏览器访问网页跑出来的数据,在关键字段层面一致**。

这个证据链的工具叫 fixtures + replay + diff。

## 三层架构

```
真实网页 / App                      我方 adapter / 接口
    │                                       │
    │ 1. Record (人工 + CloakBrowser)        │
    ▼                                       │
fixtures/snapshots/                         │
  GET_x.req.json    ── 2. Replay ──────────►│
  GET_x.resp.json   ◄── 3. Diff ────────────┘
  GET_x.meta.yaml
    │
    ▼
fixtures/reports/<date>-replay.md  + trend.json
    │
    ▼
CI Schedule → 一致率 < 90% → Issue
```

## 文件格式

见 `站点经验库/_templates/fixtures/_README.md`。核心:

| 文件 | 角色 |
|---|---|
| `snapshots/<METHOD>_<slug>.req.json` | 请求快照(method/url/headers/body) |
| `snapshots/<METHOD>_<slug>.resp.json` | 响应快照(status/headers/body) |
| `snapshots/<METHOD>_<slug>.meta.yaml` | endpoint 语义名 / expiry / 豁免字段 / 容忍度 |
| `recordings/<date>-session.har` | 原始 HAR 抓包(30 天后删) |
| `recordings/<date>-session.jsonl` | CloakBrowser CDP 录制(30 天后删) |
| `reports/<date>-replay.md` | 重放报告 |
| `reports/trend.json` | 一致率历史 |
| `actual/<prefix>.actual.json` | replay 产出(gitignore,本地诊断用) |

## 操作流程

### Step 1 录制(首次)

二选一:

**A. 用 CloakBrowser**(推荐反爬严的站点)
```bash
pip install cloakbrowser && python -m cloakbrowser install
python tools/recorder/cloak_recorder.py \
    --domain thaiairways.com \
    --start-url https://www.thaiairways.com/ \
    --duration 180
python tools/recorder/recording_to_fixtures.py \
    --recording 站点经验库/thaiairways.com/fixtures/recordings/<date>-session.jsonl \
    --domain thaiairways.com --apply
```

**B. 用浏览器手抓 HAR**(任何站点)
- Chrome DevTools → Network → 右键 → Save all as HAR with content
- 存到 `站点经验库/<domain>/fixtures/recordings/<date>-session.har`
```bash
python tools/recorder/har_to_fixtures.py \
    --har 站点经验库/<domain>/fixtures/recordings/<date>-session.har \
    --domain <domain> --apply
```

### Step 2 人工 review meta.yaml

每个 `.meta.yaml` 必填字段:`endpoint` / `category` / `sensitive` / `requires_auth`。
- `category`: `public-read` / `search` / `detail` / `list` / `session` / `config`
- **不允许** `payment` / `order-create` / `pay-confirm`(红线)
- 含 PII 的响应:`sensitive: true`
- 业务字段加 `tolerance` 容忍度规则(见 field_rules.py)

### Step 3 重放

```bash
python tools/replayer/snapshot_replay.py \
    --domain thaiairways.com \
    --target https://my-adapter.local:8080
python tools/replayer/consistency_report.py \
    --domain thaiairways.com \
    --threshold 0.90
```

### Step 4 看报告

`站点经验库/<domain>/fixtures/reports/<date>-replay.md`,关注:
- 结构 PASS 比例:< 100% → 字段缺失/类型错(立即修)
- 整体一致率:< 90% → 业务字段对不上(看 diff 明细)
- 与上次趋势对比:`trend.json` 看历史

## 阈值

| 一致率 | 状态 | 含义 |
|---|---|---|
| ≥ 95% | PASS | 接口可发版 |
| 90-95% | WARN | 接口可用,但要看 diff 明细决定是否标 known-failures |
| 80-90% | FAIL | 必须修后再发 |
| < 80% | CRITICAL | 接口疑似失效,立即 hold |

CI 默认阈值 `0.90`,通过 `workflow_dispatch` 输入可调。

## 字段豁免

默认豁免(`tools/replayer/field_rules.py` 的 `VOLATILE_FIELDS`):
- 标识类: sessionId / requestId / uuid / id
- 时间类: timestamp / ts / time / createdAt / updatedAt
- 安全类: token / jwt / csrf / nonce / etag
- 追踪类: traceId / spanId / x-request-id
- 缓存类: x-cache / expires

每个 endpoint 的 `meta.yaml` 可追加 `volatile_fields` 列表,本接口额外豁免。

## 容忍度规则

业务字段允许漂移的(`field_rules.py` 的 `TOLERANCE_RULES`):

| 规则名 | 含义 |
|---|---|
| `exact` | 严格相等(默认) |
| `percent_5` / `percent_10` / `percent_20` | 数值差异 < N% |
| `int_diff_5` / `int_diff_20` | 整数差 ≤ N |
| `list_len_10pct` / `list_len_20pct` / `list_len_50pct` | 列表长度差异 < N% |
| `ignore` | 永远通过(慎用) |

`meta.yaml` 的 `tolerance` 字段映射 jsonpath → 规则名:
```yaml
tolerance:
  "data.flights[].price": percent_10
  "data.flights[].availability": int_diff_5
```

## CI

`.github/workflows/consistency-replay.yml`:
- pull_request: 验 fixtures schema + 评分(不跑 replay)
- schedule: 每周二 UTC 02:23 跑 replay + diff,< 90% 开 issue
- workflow_dispatch: 手动跑,可传 adapter URL 和阈值

job 拆两个:
- `validate-schema`: schema 检查 + 评分(总是跑)
- `replay-diff`: 真实重放(只 schedule 和带 adapter_url 的手动触发跑)

## 红线

1. **支付 / 真实下单 / 改密码接口绝对不录** — 违规直接删
2. **PII 响应必须标 sensitive: true** — CI 不打印
3. **fixtures 超 30 天必须重录** — `expires_at` 强制
4. **CI 跑不通真实抓取** — 只重放本地 fixtures,首次录制要人在本地配合
5. **一致率刷分禁忌** — 不允许把所有字段加 `ignore` 规则刷高一致率,review 时盯防

## 与 known-failures.md 的关系

- known-failures: **流程类**失败(WAF / 路由 / 会话 / 支付边界)
- fixtures + reports: **数据类**失败(字段对不上)

一致率长期 < 90% 的失败,要把根本原因写进 `known-failures.md`,加 eval。

## 评分

`score_skills.py` 升级到四段分(v0.3.6):

| 维度 | 满分 | 一致性细则 |
|---|---:|---|
| 结构 | 25 | |
| 实战 | 25 | |
| **一致性** | **30** | fixtures 存在 5 + ≥ 3 snapshot 5 + expiry 有效 5 + 30 天内报告 5 + 一致率分档 10 |
| 漂移 | 20 | |

**无 fixtures → 一致性 0 → 总分上限 70**。结构再完美也只能是"结构就绪,缺一致性验证"。
