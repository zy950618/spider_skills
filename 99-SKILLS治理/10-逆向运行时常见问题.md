# 逆向运行时常见问题

> 横切方法论文档。被 17 个 execution skill 在 Tool Policy 区段引用，集中沉淀"运行时"层面踩过的坑：断点 / 时间 / cookie / TLS 指纹 / 风控恢复 / 接口版本漂移。

本文不是 SKILL.md，不会被自动触发；它是 execution skill 共享的"运行时 checklist"。遇到对应症状时，先读这里再决定怎么处理。

## E1. 断点处理

### 症状
- 浏览器 DevTools 一打开 `Sources` 面板，页面立刻报 `debugger` 死循环、CPU 100%、tab 卡死
- 加了 XHR / Fetch 断点后，请求触发但断点位置在 polyfill / minified 代码深处，找不到业务入口
- 在 IDA / x64dbg 上下软件断点（INT3）后程序立即崩溃或走异常处理分支
- Frida hook 上去之后目标进程退出 / 输出"detected debugger"

### 检测方法
- 看页面是否反复进入 `debugger` 语句：DevTools → Sources → 勾掉 "Pause on exceptions"，看是否还卡
- 用 `setInterval(() => { const t = performance.now(); debugger; if (performance.now() - t > 100) alert('devtools open'); }, 1000);` 这种 anti-debug 模式存不存在
- 检查 `Function.prototype.toString` 是否被改写：`(()=>{}).toString.toString()` 看返回值是否是 native code
- IDA 里看 PEB.BeingDebugged / NtQueryInformationProcess 调用
- Frida：看目标进程是否检测 `/proc/self/status` 中的 TracerPid

### 应对
- **Chrome DevTools 断点选型**:
  - 条件断点（Conditional breakpoint）：业务字段已知时用，例如 `headers['x-sign'] !== undefined` 才停
  - XHR/Fetch 断点：定位"发起请求时的调用栈"，配合 `Async` 栈跟踪
  - DOM 断点（subtree / attribute modifications）：处理 SPA 路由后才注入的加密 hook
  - Event 断点（mouse click / keypress）：定位"用户操作触发的加密流程"
- **anti-debug 绕过**:
  - DevTools → Sources → 右键 `Never pause here`，把 `debugger;` 行加入黑名单
  - Override `Function.prototype.toString` 让伪装函数返回 native code 字串
  - Override `setInterval` / `setTimeout`，劫持时间差检测
- **IDA / x64dbg**: 用硬件断点（DR0-DR3）代替软件断点，硬件断点不改原指令、可以绕过 INT3 扫描
- **Frida 替代静态断点**: `Interceptor.attach(addr, {onEnter, onLeave})`，运行时 hook 不触发大多数 anti-debug；attach 前用 `Frida.gadget` 而非 frida-server 也能进一步降低检测面

### 真实案例
thaiairways 的 reese84 challenge 在生成 token 前会调用 `console.log` / `console.debug` 系列采集行为；如果先打开 DevTools，`console.*` 已经被 DevTools 重写，token 生成路径走 fallback 分支，导致即使 hook 到加密函数也拿不到正确 sign。处理方式：在打开 DevTools 之前先注入脚本 hook 住 `console.log` 等方法（保留原引用），再开 DevTools 调试。

## E2. 时间更新 / 时间漂移

### 症状
- 录制时 sign 算对、24 小时后回放 sign 错（且服务端返回 token expired / sign invalid）
- 同一份 fixtures 在不同机器跑结果不一样
- 抓包看请求 timestamp 与本地时间相差几秒，签名校验失败
- `freezegun` 冻结时间后，依赖时间的 fixture 全部断言失败

### 检测方法
- 看 `站点经验库/<domain>/fixtures/<endpoint>/meta.yaml` 的 `expires_at` 字段：过期就该重录
- 抓两次请求对比 timestamp 字段精度：秒（10 位）/ 毫秒（13 位）/ 微秒（16 位）
- `curl -I` 看响应头 `Date:` 与本地 `date -u` 差几秒
- 在 sign 函数处下断点，打印 `Date.now()` 实际被调用时的值

### 应对
- 签名包含 timestamp 时，snapshot 的 `meta.yaml` 必须写 `expires_at`，CI 检查超过该时间就重录而不是直接 diff
- 客户端与服务端时间对齐：用响应头 `Date:` 作为参考，本地时间偏差超过 N 秒时校准
- 精度对齐：先看抓包，sign 拼接的 timestamp 到底是 `Date.now()`（毫秒）还是 `Math.floor(Date.now()/1000)`（秒）；写死在算法实现里
- 测试环境用 `freezegun` 冻结时间时，**只冻结业务时间，不冻结 TLS 握手 / cookie 过期判断**；或者把所有 fixture 的 timestamp 也按冻结点回写

### 真实案例
thaiairways 的 sign 包含 `(Date.now() / 1000 | 0).toString()`，精度是秒。第一版 Python 实现直接用 `int(time.time() * 1000)` 拼了 13 位毫秒值，sign 全错。修复后改成 `str(int(time.time()))`，并在 unit test 里用 `freezegun.freeze_time("2026-05-23T10:00:00Z")` 锁定时间复现固定 sign 值做断言。

## E3. cookie 清理

### 症状
- 同一份请求脚本，第一次跑通、第二次 403；或反过来
- 团队成员 A 跑通，团队成员 B 用同样代码 403
- 长跑批量任务前 100 个 OK、第 101 个开始全 403
- 复制别人的 HAR 文件本地重放，第一次 OK 后再跑全部失败

### 检测方法
- 对比两次请求的 `Cookie:` 头，看 reese84 / incap_ses / ak_bmsc 这类风控 cookie 是否相同
- 长效 cookie（认证态：JSESSIONID / auth_token，TTL 小时/天级）vs 短效 cookie（风控指纹：reese84 形如 `3:xxxx==:xxxxx...`，长度 32+ base64-like，TTL 分钟级）
- Playwright/Puppeteer：`context.cookies()` 看每次 case 之前 cookie jar 是否为空
- HAR 文件用 `jq '.log.entries[].request.cookies'` 看是否携带历史 cookie

### 应对
- 测试前清空 Profile 优于手工 `clear cookies`：Profile 包含 LocalStorage / IndexedDB 里的指纹缓存
- 区分两类 cookie：长效的认证态可以复用，短效的风控指纹**每 case 重置**
- Playwright：每个 test 用独立的 `browser.newContext()`，而不是共享 context
- HAR 重放前用脚本剥离 `cookies` 字段，让首次请求自然 set cookie
- 写"脏 cookie 排查路径"：先 `clear-site-data` → 再无痕模式 → 再换浏览器版本

### 真实案例
thaiairways 批量测试 booking 200 单时，前 80 单稳定、第 81 单起开始 403；排查发现 reese84 cookie 在 server 端有"同 cookie 高频访问"判定，需要每 50 请求重置一次 cookie（重新跑 challenge 生成新 reese84），并把这条写进 `站点经验库/thaiairways.com/known-failures.md`，标 `symptom=batch_403_after_n_requests / marker=cookie_reuse_throttle`。

## E4. TLS / 请求指纹

### 症状
- 浏览器能访问、Python `requests` 直接 403 / 521
- 同一份 headers 用 `curl` 通、用 `requests` 不通
- Node `axios` 通、Python `requests` 不通
- 加了完整 headers / cookies 还是 403

### 检测方法
- 用 `https://tls.peet.ws/api/all` 或 `https://tools.scrapfly.io/api/fp/anatomy` 对比：浏览器 ja3_hash vs `requests` ja3_hash
- 看响应是否带 `cf-ray` / `x-iinfo` / `x-cdn` 标识，判断走 Cloudflare / Imperva / Akamai
- `requests` 默认 ciphers 顺序、TLS 版本（1.2 vs 1.3）、ALPN（http/1.1 vs h2）与真 Chrome 都不同

### 应对
- **Python 工具选型**:
  - `curl-cffi --impersonate chrome120`（推荐，覆盖 JA3+JA4+HTTP/2 帧顺序+H2 setting 指纹）
  - `tls-client`（Go 移植，灵活但维护稍弱）
  - `httpx[http2]` + 自定义 ciphers（最轻量但还原度低）
- **Node**: `undici` + 自定义 client / `node-fetch` + agent 改 ciphers / Puppeteer headless 兜底
- **验证**: 把 `curl-cffi` 请求打到 `https://tls.peet.ws/api/all`，对比 `ja3_hash` 是否与真 Chrome 一致
- 仍不通时检查 HTTP/2 帧顺序（SETTINGS / WINDOW_UPDATE / HEADERS 的发送顺序），Imperva / Akamai 会校验

### 真实案例
thaiairways 初接入时 `requests` 全部 403（Imperva 拦截），切到 `curl_cffi.requests.get(url, impersonate="chrome120")` 后立即 200。验证：peet.ws 给出的 ja3_hash 从 `requests` 的 `b32309a26951912be7dba376398abc3b` 变成与本机 Chrome 120 完全一致的 hash 后才稳定。该结论写进 `站点经验库/thaiairways.com/known-failures.md` 的 `tls_fingerprint_rejected` 条目。

## E5. 频控 / 风控触发后的恢复策略

### 症状
- 单次请求 OK，并发 / 高频后开始 403、429、503 混合返回
- 同 IP 累计 N 次后全部失败，换 IP 立刻恢复
- 同账号高频触发"账号风险"提示，需短信验证
- 间隔几小时再跑又能用了

### 检测方法
- **风控 vs 业务错的判别**:
  - 风控：HTTP 403 / 429 / 503 + reese84 / incap_ses cookie 被重置 + 响应体是 challenge HTML
  - 业务错：HTTP 200 + JSON 里 `errorCode` 字段（库存 / 价格 / 航线无效等）
- 看响应头是否带 `retry-after`、`x-ratelimit-remaining`、`cf-ray`
- 监控 ja3 是否被 server 临时拉黑（同 ja3 切 IP 也不通）

### 应对
- **触发风控后选项**（按代价从低到高）:
  1. 降速（限频）：QPS 降一档，加 jitter
  2. 重新 challenge：重跑 reese84 / incapsula 拿新 cookie
  3. 换 IP（代理池）：住宅 IP > 数据中心 IP
  4. 换账号（账号池）：仅在需要登录态时
- **指数退避 + jitter**: `delay = base * 2^attempt + random(-jitter, +jitter)`；`base=1s, max=60s, jitter=200ms`
- **长跑批量任务的中段失败恢复**:
  - 断点续传：每完成 N 单 flush 到磁盘 `progress.jsonl`
  - 去重：用 booking_id / search_id 建本地 sqlite 表，重启后跳过已完成
  - 失败重试队列单独走，不阻塞主流程

### 真实案例
thaiairways 批量 search 在 100 QPS 时触发限流（5 分钟内 ~30% 请求返回 403 + reese84 cookie 被重置）。降到 10 QPS + 每请求间 `time.sleep(random.uniform(0.8, 1.2))`（即 jitter ±200ms）后稳定运行 8 小时无中断。该参数固化进 `站点经验库/thaiairways.com/notes.md` 的 rate-limit 段，并在 adapter.yaml 的 `throttle` 字段写死 `qps=10, jitter_ms=200`。

## E8. 接口版本变更检测

### 症状
- 跑了 N 周的脚本突然部分字段拿不到 / 拿到 `null`
- fixtures 一致性从 100% 降到 95%，diff 显示新增了若干 `extra` 字段
- 业务端反馈"某些订单类型识别不出来"（其实是后台加了新枚举值）
- 接口路径 / 参数名静默变更（前端缓存了旧 JS bundle 没察觉）

### 检测方法
- 用 `tools/replayer/snapshot_diff.py` 跑 fixtures，看 `mismatched` / `missing` / `extra` 三段
- **检测项**:
  - 字段增（actual 比 snapshot 多）→ `extra` 段
  - 字段删（snapshot 比 actual 多）→ `missing` 段
  - 字段类型变（string → int）→ `mismatched` 段 `reason="type mismatch"`
  - 字段值域变（枚举值新增）→ `mismatched` 段 `reason="value mismatch"`，需结合业务判定
  - 嵌套层级变化 → 通过 path 前缀对比识别（`a.b.c` vs `a.b.d.c`）
- 用 `tools/replayer/schema_alert.py`（本次新增）对 diff 结果二次分类：BREAKING vs COMPATIBLE

### 应对
- **预警机制**: CI 定时跑（每周 / 每月），输出报告到 `站点经验库/<domain>/fixtures/reports/<date>-schema-diff.md`
- **基线管理**:
  - 抓包后初次接入 → 录第一份 snapshot 作基线，meta.yaml 标注 `baseline=true`
  - 后续每 N 天跑一次 diff，差异写报告但不直接覆盖基线
  - 字段消失时判断是接口下线还是改名：查调用频次（grep 业务代码 + 日志）+ 业务方文档
- **分级处理**:
  - BREAKING（字段删 / 类型变 / 嵌套结构变 / required 消失）→ CI 失败，阻断发布
  - COMPATIBLE（字段新增 / 枚举值新增 / optional 消失）→ CI warning，写报告不阻断

### 真实案例
thaiairways 在 2026-04 的某次 release 后，`flightLegs[].cabinClass` 字段新增了枚举值 `CODESHARE_PARTNER`（原仅 `Y/W/C/F`）。`snapshot_diff.py` 标 mismatched，但 `schema_alert.py` 识别为 COMPATIBLE（枚举值新增、字段类型未变），输出 warning 报告到 `站点经验库/thaiairways.com/fixtures/reports/2026-04-12-schema-diff.md`，不阻断 CI。同 release 的另一处 `paymentMethods.providers` 从 list 变成 dict，`schema_alert.py` 识别为 BREAKING，CI 失败并 @ owner 排查，最终改 adapter 兼容。
